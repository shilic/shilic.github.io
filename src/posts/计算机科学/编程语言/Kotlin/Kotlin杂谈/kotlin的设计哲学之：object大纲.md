Kotlin的设计哲学之：object与单例模式》修订版大纲

开场故事：Java 单例的繁琐 vs Kotlin 一行 object。
引出核心问题：为什么 Kotlin 能用语言特性消灭一个设计模式？
预告文章结构：从历史包袱到语言设计，再到跨语言对比和关键字选择哲学。
2. 传统单例模式的 Java 实现及其缺陷（~600字）
三种经典写法（饿汉、懒汉同步、双重检查锁）及各自问题。
反射、序列化对单例的破坏。
总结：单例本身简单，但 Java 缺乏语言级支持，被迫手写样板代码。
3. Kotlin object的机制与编译真相（~800字）
基本语法：object MySingleton
反编译揭示：饿汉式、INSTANCE静态字段、final class、私有构造器。
线程安全由类加载保证。
与 Java 枚举单例的相似性和优势（可继承接口、更灵活）。
4. 【新增】与其他语言单例实现的对比（~500字）
JavaScript 的单例：
传统方式：模块模式（IIFE + 闭包）、ES6 模块（export default {}天然单例）、class+ 静态 getter。
特点：JS 没有类加载机制，单例依赖于模块缓存或闭包，本质上是“对象字面量”或“模块实例”。
与 Kotlin object的异同：
相同点：都试图用最简单的语法表达“唯一实例”。
不同点：Kotlin 利用 JVM 类加载保证线程安全和唯一性；JS 依赖运行时模块系统，且没有真正的私有构造器。
启示：两种语言都朝着“语言内置单例”的方向演进，但实现路径受限于底层平台。
可选：简要提及其他语言（如 Swift static let、Python __new__）以拓宽视野。
5. 设计哲学分析：简洁、安全、零样板（~600字）
简洁优先：一行 object替代十几行 Java。
安全优先：不可继承、私有构造、线程安全。
零样板：Kotlin 用语言特性消灭了“单例模式”这个模式本身。
引用 JetBrains 理念：语言提供工具，而非模式。
6. 【新增】为什么是 object而不是 Singleton？——关键字命名的哲学（~500字）
直观疑问：既然主要用途之一是单例，为什么不叫 singleton？
深入分析：
object在 Kotlin 中代表“一个对象实例”，它不仅仅用于单例，还用于：
匿名内部类（object : Listener）
伴生对象（companion object）
对象表达式（临时实例）
所有这些场景的共同点是：声明一个具体的、有名字或匿名的对象，而非声明一个类。
如果用 singleton，则无法涵盖匿名对象和伴生对象，会导致语言中出现两个含义重叠的关键字（singleton和 object），增加复杂度。
设计思路：Kotlin 追求一致性——object就是“对象字面量”，与类声明（class）形成互补。单例只是 object的一种特殊形式（顶层具名对象）。
对比 Java：Java 用 class声明类，然后用各种技巧获得唯一实例；Kotlin 直接说“我要一个对象”，语义更直接。
结论：关键字的选择反映了 Kotlin 对“对象”的一等公民地位，而非仅仅服务于单例模式。
7. object的多种形态与适用场景（~500字）
顶层 object：纯单例，适合无状态工具类。
companion object：替代静态成员，注意它不是全局单例，而是依附于外部类的独立实例。
匿名 object：一次性实例，非单例。
object expression：每次求值产生新实例（除非赋值给 val 且只执行一次）。
何时不该用 object：需要懒加载、传参构造、继承、mock 测试。
8. 与依赖注入框架的协作（~400字）
object作为单例工厂 vs DI 容器的单例管理。
局限性：无法构造函数注入，只能手动 setter 或 lateinit。
最佳实践：无状态工具类用 object；有依赖的服务交给 DI。
9. 争议与陷阱（~500字）
序列化破坏单例（需 readResolve）。
测试困难（final class 难以 mock）。
内存泄漏风险（饿汉式，即使从未使用也占用资源）。
继承限制（只能实现接口，不能继承类）。
反思：Kotlin 选择了最简单安全的默认实现，trade-off 留给开发者按需处理（如用 by lazy模拟懒加载）。
10. 结论（~300字）
重申核心：Kotlin 通过 object将单例提升为语言特性，体现了“约定优于配置”和“最小惊奇”。
关键字命名智慧：object的统一性远超单例本身，是 Kotlin 面向对象哲学的缩影。
结尾：最好的单例是忘记它是单例，最好的语言设计是让常见模式消失。
